# Qodly Custom Component — Internationalization (i18n)

## Studio settings: translatable attributes

Edit the component’s `*.settings.ts` (for example `src/components/AppointmentSchedule/AppointmentSchedule.settings.ts`).

1. Import `getStaticFeaturesExperimentalFlag` from `@ws-ui/webform-editor` (same barrel as `ESetting` / `TSetting`).
2. Derive whether i18n is enabled for the environment (matches how the standalone editor wires `I18nProvider`):

```ts
import { ESetting, TSetting, getStaticFeaturesExperimentalFlag } from '@ws-ui/webform-editor';

const isI18nEnabled = getStaticFeaturesExperimentalFlag('i18n');
```

3. For any setting whose value should be translated in Studio, use the i18n field type when i18n is on, and fall back to a plain text field otherwise:

```ts
{
  key: 'myLabel',
  label: 'My label',
  type: isI18nEnabled ? ESetting.I18NFIELD : ESetting.TEXT_FIELD,
  defaultValue: '',
},
```

`ESetting.I18NFIELD` is the setting type that stores structured i18n keys/values; `ESetting.TEXT_FIELD` is the non-i18n fallback.

## Renderer (and build): resolved text for the current language

Inside `*.render.tsx` (or `*.build.tsx` when you need the same behavior on canvas), use `useI18n()` from `@ws-ui/webform-editor`:

```tsx
import { useI18n } from '@ws-ui/webform-editor';

const { i18n } = useI18n();
const lang = i18n?.userLang?.primary ?? 'de';
const entry = i18n?.keys?.[yourKeyName];
const value = entry?.[lang] ?? entry?.de;
```

- `yourKeyName` is the i18n key string you store in props (often aligned with what the `I18NFIELD` setting persists).
- `lang` is the active user language; `i18n.userLang.primary` is the usual source. Adjust if your component receives locale from props instead.
- `entry?.de` is a common fallback when the current language has no value yet.

`useI18n` must run under the editor’s `I18nProvider` (Qodly Studio / standalone setup). If it throws, the component is outside that provider.
